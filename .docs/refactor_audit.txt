  Critical Bug                                                                                                                                                                                                                                                                                                
                                                                                                                                                                                                                                                                                                              
  CombatEncounter.victory_scene / defeat_scene are dead fields — High                                                                                                                                                                                                                                         
                                                                                                                                                                                                                                                                                                              
  views.py:274 and views.py:305 pass enemy.victory_scene and enemy.defeat_scene to resolve_combat_end. But the quest builder's update_combat_encounter (quest_builder.py:625-629) saves routing to CombatEncounter.victory_scene / CombatEncounter.defeat_scene. These two fields are written but never read  
  by the game engine. Any routing set via the canvas UI is silently ignored at runtime. An enemy can only have one victory/defeat destination globally, making the per-encounter routing UI a no-op.                                                                                                          

  ---
  File-by-File Findings

  game/views.py

  choice_resolve is a fat view — Medium
  Lines 98–196. The RivalClaim query at line 163 (RivalClaim.objects.filter(player_property__session=session, resolution_scene=next_scene)) is business logic — deciding whether an active contest resolves — executed directly in the view. This belongs in
  property_service.resolve_contest_if_applicable(session, next_scene).

  Inline imports throughout — Low
  choice_resolve:129, scene_detail:82, scene_panel:434-439, choice_panel:675-678, choice_create:736-738, choice_requirements_save:836-840 all have from .models.xxx import Xxx mid-function. None of these trigger circular imports; they should all be at the top of the file.

  scene_items_save fetches Scene twice — Low
  scene_items_save:588 calls get_object_or_404(Scene, pk=scene_id, quest_id=quest_id) and discards the result, then scene_combat_save:637 does Scene.objects.get(pk=scene_id) again. Same pattern in scene_combat_save:624–637. The first fetch should be reused.

  choice_create copy-pastes choice_panel context-building — Medium
  Lines 740–771 are near-identical to choice_panel:668–718. Any future change to the choice panel context must be made in two places.

  No ownership check on choice_resolve — Medium (security)
  choice_resolve:107 fetches any Choice by PK from the URL. There's no check that the choice belongs to session.current_scene. A player can POST to /game/choose/<any_id>/ and execute choices they've never seen, bypassing requirement gates and jumping to arbitrary scenes.

  game/services/combat.py

  get_or_create_combat_state writes EventLog — Medium
  Line 86–88: EventLog.objects.create(session=session, text=f"You square up against ..."). The documented contract is that services return log strings and views write them. This is the one service that breaks that contract.

  resolve_combat_end writes EventLog and returns render context — High
  Lines 108, 112, 116, 118 write directly to EventLog. Line 122–125 calls build_render_context and returns a template context dict. This couples the service to both the DB-write pattern (breaking the log contract) and HTTP rendering (making it impossible to call outside a request cycle). Both issues
  coexist in the same function.

  game/services/session.py

  build_render_context belongs in the view layer — Medium
  This assembles a template context dict — it is view work. Its placement in the service layer caused the circular import that forces the from .scene import ... inline import at line 38. Moving it to a views.py helper would remove that circular dependency and correctly locate the HTTP-concern code.

  game/services/scene.py

  N+1 in get_available_choices — Medium
  For each choice: choice.requirements.exists() (line 35) is one query; rg.evaluate(ctx) for rg in choice.requirements.all() (line 38) is another; and inside RequirementGroup.evaluate, self.requirements.all() (requirements.py:132) is yet another per-group. A scene with 5 choices each having 2
  requirement groups and 3 requirements each = 5 + 5×2 + 10×1 = 25 queries just for choice filtering.

  Fix: scene.choices.prefetch_related('requirements__requirements').select_related('quest').

  get_notice_board prefetch wasted — Medium
  Line 108 uses prefetch_related('requirements__requirements'), but line 117 then calls quest.requirements.exists() which bypasses the prefetch cache and fires a new query. Use if quest.requirements.all(): or check len(quest.requirements.all()) against the already-fetched prefetch.

  game/services/property_service.py

  process_turn_income re-executes queryset — Low
  Line 37: if properties.exists(): is called after for pp in properties: has already evaluated the queryset. Use if logs: instead.

  check_rival_contests evaluates queryset twice — Low
  Line 62: contestable.exists() then line 67: random.choice(list(contestable)) fires the queryset a second time. Materialize to a list once: contestable_list = list(contestable) and check if not contestable_list:.

  game/services/quest_builder.py

  update_scene allowed_fields check is a dead abstraction — Low
  Lines 429–431: allowed_fields is defined but the pattern if 'title' in allowed_fields: is checking a hardcoded string against a hardcoded set — it's always True. The set has no effect. Either use it as an actual guard (e.g., filter by keys present in data) or remove it.

  delete_scene nulls CombatEncounter outside transaction — Medium
  Lines 503–504 (CombatEncounter.objects.filter(...).update(victory_scene=None/defeat_scene=None)) run before the transaction.atomic() block at line 506. If the atomic block fails (e.g., cascading delete fails), the encounter routing has already been cleared non-atomically. Move lines 503–504 inside
  the with transaction.atomic(): block.

  build_requirement_groups_from_post leaks orphaned records — High
  Line 657: obj.requirements.clear() removes M2M links but does not delete the RequirementGroup or Requirement objects. Every save creates fresh records. After N saves of a choice's requirements, the DB has N×groups + N×requirements orphaned rows with no foreign key references. Add:
  old_groups = list(obj.requirements.all())
  obj.requirements.clear()
  Requirement.objects.filter(groups__in=old_groups).delete()
  RequirementGroup.objects.filter(pk__in=[g.pk for g in old_groups]).delete()

  game/services/progression.py

  LEVEL_UP_FLAVOR indexed by level - 2 — Low
  progression.py:57 uses LEVEL_UP_FLAVOR[new_level - 2]. Same in combat.py:118. A list indexed by level - 2 means level 2 = index 0, level 7 = index 5. If MAX_LEVEL changes or an out-of-range level is reached, this silently produces wrong text or raises IndexError. Use a dict keyed by level number:
  LEVEL_UP_FLAVOR = {2: "...", 3: "...", ...}.

  game/models/world.py

  Scene.roll_stat unconstrained — Medium
  roll_stat = models.CharField(max_length=50, blank=True) accepts any string. Only strength, agility, intellect, charisma are valid. Should have choices=[(v, v) for v in STAT_FIELD_MAP.values()] and a migration. Without this, a typo in the quest builder silently produces a stat modifier of 0 (via
  getattr(effective_stats, 'typo', 10) in scene.py:9).

  Scene.ending_type allows blank on ending scenes — Medium
  An ending-type scene with ending_type='' passes the if next_scene.is_ending check in maybe_complete_quest but then CompletedQuest is created with ending_type='' and XP_AWARDS.get('', 0) returns 0 XP. No DB constraint prevents this.

  Choice.resolve_target is defined but never called — Low
  world.py:200-213. All routing goes through resolve_roll in services/scene.py or direct choice.target_scene access. This method is dead code.

  game/models/player.py

  CompletedQuest missing unique_together — High
  maybe_complete_quest uses a TOCTOU check: .filter(...).exists() then .create(...). Two simultaneous requests to the same ending scene can both pass the exists() check and create duplicate completions, awarding double XP. Add unique_together = ('session', 'quest') and handle IntegrityError in
  maybe_complete_quest.

  game/models/requirements.py

  Requirement fields have no cross-field validation — Low
  A stat_gte requirement with blank stat_name silently evaluates getattr(ctx.stats, '', 0) >= 0 — always True. A has_item with null required_item_id evaluates None in ctx.inventory — always False. Add clean() validation or at minimum a db_constraint.

  game/models/property.py

  PlayerProperty.upgrade_tier is dead — Low
  Line 26: upgrade_tier = models.IntegerField(default=1) with comment "reserved — no logic". Dead field written to DB on every session. Remove until actually needed.

  resolution_scene duplicated on Property and RivalClaim — Medium
  Property.resolution_scene and RivalClaim.resolution_scene are separate FK fields. At claim creation (property_service.py:72) the property's value is copied to the claim. If the property's resolution scene later changes, existing claims keep stale routing. Pick one source of truth: remove
  Property.resolution_scene and always route through RivalClaim.resolution_scene, set at claim creation time.

  game/models/combat.py

  CombatEncounter.victory_scene / defeat_scene are dead — High
  See critical bug section above. These fields are written via the quest builder but read from Enemy at runtime. The fields can be removed from CombatEncounter, or combat_attack must be fixed to read them from the encounter instead of the enemy.

  game/tests.py

  Hardcoded fixture PKs — Medium
  test_stat_gated_choice:50: Choice.objects.get(pk=5). test_combat_victory_and_quest_completion:257: self.assertEqual(self.session.current_scene.pk, 22). These break silently if fixture data is reloaded or re-ordered. Look up by key: Choice.objects.get(scene__key='...', label__icontains='...') and
  Scene.objects.get(key='...').

  No tests for:
  - Property system (process_turn_income, check_rival_contests, resolve_contest)
  - Flag effects on choices
  - SceneUnlock / complete_scene
  - get_effective_stats passive bonuses
  - award_xp multi-level crossing

  core/settings.py

  Hardcoded SECRET_KEY — Medium (security)
  Line 23: SECRET_KEY = 'django-insecure-...'. Fine for dev but should be sourced from an env variable before any deployment.

  ---
  Cross-Cutting Issues

  1. EventLog write contract inconsistency
  combat.py services write EventLog.objects.create() directly. All other services return log strings for the view to write. The pattern is documented as intentional in session.py but violated in combat.py. This means resolve_combat_end can't be tested without a full DB + a real request cycle.

  2. N+1 in every requirement evaluation
  Every call to get_available_choices or get_notice_board fires one DB query per choice × one query per requirement group × one query per requirement. The fix is prefetch_related('requirements__requirements') on every queryset that fetches choices or quests, and replacing .exists() with
  len(prefetched_list) > 0.

  3. build_render_context in service layer
  This function assembles HTTP template context (HTTP concern) and imports from scene.py circularly. Moving it to views.py as a private _build_context(...) helper resolves the circular import, correctly locates the concern, and lets the service layer stay pure.

  4. RequirementGroup orphan accumulation
  Every quest builder save permanently leaks rows. In a heavily-used builder this will silently grow the DB. The orphan cleanup in build_requirement_groups_from_post is a one-line fix.

  Missing Abstractions Worth Extracting

  1. resolve_choice_transition(session, choice, stats, inventory, completed_map) — extract the 98 lines of choice_resolve into a service function that returns (next_scene, log_entries, turn_summary). The view calls it and writes logs.
  2. GameSessionPK_SESSION_KEY = 'game_session_id' constant — this string is used in 6+ view functions directly and is not in constants.py.
  3. prefetch_choices_with_requirements(scene_qs) — a queryset helper that applies the correct prefetch_related chain. Used in get_available_choices and scene_detail.

  ---
  What's Working Well — Preserve This

  - PlayerContext dataclass — clean, testable, no ORM dependency. Good separation.
  - EffectiveStats dataclass — correctly separates mutable DB stats from derived display values.
  - get_effective_stats passive bonus pattern — clean, doesn't mutate PlayerStats.
  - Admin → View delegation pattern — QuestAdmin.get_urls() delegates to view functions wrapped with self.admin_site.admin_view(). Keeps auth in one place without code duplication.
  - award_scene_items in-place dict mutation — returning awarded items and updating the caller's inventory dict avoids stale reads without an extra DB round-trip.
  - validate_quest in quest_builder.py — comprehensive, covers orphans, routing gaps, combat misconfiguration, hub assignment. A solid authoring guard.
  - build_requirement_groups_from_post — single function owns all POST→model parsing for requirements. Good encapsulation despite the orphan leak bug.
  - flags.py — small, focused, uses update_fields=['flags'] for minimal writes.
